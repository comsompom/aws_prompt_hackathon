How to Submit Your Prompt
Submission Window: March 1 - June 1, 2026 (90 days)

What You'll Submit
Your submission should be a production-ready prompt that helps developers accomplish real AWS tasks. Think of it as contributing to a shared knowledge base that thousands will use to build their applications.

Submission Requirements
When you create your BUIDL, you'll provide:

1. Your Complete Prompt (in the Details section) Include the full, verbatim prompt exactly as a developer would use it with an AI assistant. This should be copy-paste ready—not a description of what the prompt does, but the actual prompt itself.

Example of what we're looking for:

✅ "Create a production-ready EKS cluster with the following specifications: Use Karpenter for node provisioning with Spot instance support, implement KEDA for workload-based autoscaling, configure AWS Load Balancer Controller for ingress..."
❌ "A prompt that helps set up Kubernetes clusters"
2. Context & Documentation Help others understand when and how to use your prompt:

Prerequisites: What does someone need before using this? (AWS account setup, specific permissions, existing resources)
Use Case: What problem does this solve? Who is this for?
Expected Outcome: What will developers have after running this prompt?
Troubleshooting Tips: Common issues and how to resolve them
3. AWS Services & Best Practices

Which AWS services does your prompt utilize?
How does it align with the AWS Well-Architected Framework?
What security, cost optimization, or operational excellence principles does it incorporate?
Quality Standards
Winning prompts share these characteristics:

Clear & Actionable – Anyone following your prompt should know exactly what to do at each step. Include specific parameter values, configuration examples, and decision points.

Production-Ready – Your prompt should guide users to build something they can actually deploy. Security groups configured properly. Monitoring enabled. Cost controls in place. Not just "hello world" demos.

Well-Documented – Think about the developer using this at 2 AM trying to solve a problem. What context do they need? What might go wrong? What should they check?

Best Practice Aligned – Follows AWS recommendations for security, reliability, performance, cost optimization, and operational excellence.
